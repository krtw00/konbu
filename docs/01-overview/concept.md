# 骨子: システム手帳のデジタル化

> **Status**: Active | 最終更新: 2026-05-21
>
> このドキュメントが konbu の骨子の正本。README / 設計 docs (`summary.md` / `goals.md` / `principles.md` / `scope.md` 他) の旧フレーミング (「text-first personal information workspace」「個人 AI ライフログ」) は本骨子に reconcile 済み (KONBU-22)。

---

## 一言で

konbu = **AI 執事が世話してくれる、自分だけのデジタルシステム手帳**。

表の顔は「システム手帳」、中身のエンジンは AI / MCP / CLI。他人とは一切共有しない一冊。

---

## 判定基準 (= 機能の取捨はこれで測る)

システム手帳という"物"が持つ性質を判定軸にする。新機能・既存機能はこの 5 つに照らして残す/落とすを決める。「シニアエンジニアに過剰と言われないか」ではなく「システム手帳に綴じるリフィルとして筋が通るか」で判断する。

1. **リフィル式** — 必要なページだけ自分で選んで綴じる (構成の自由)
2. **一冊集約** — 全部そこにあり、横断して引ける
3. **個人専用** — 他人と共有しない、自分だけの一冊
4. **スケジュールが背骨** — 時間軸 (マンスリー/ウィークリー/デイリー) が中心
5. **自分でフォーマット** — 方眼・家計簿・健康記録など、決まった枠に書く

### inbound / outbound 原則

> konbu (一冊の手帳) には **全部が流れ込む** が、**外には一切漏れない**。

「個人専用 + 一冊集約」を一文で体現する。外向き経路 (共有 / 公開 / outbound feed) は削り、内向き経路 (Google カレンダー取り込み等) は足す。

---

## AI / MCP / CLI の位置づけ

「手帳を書く・引く **執事**」として残す。これが差別化の核。MCP / CLI は「個人が自分のデータを操作する経路」なので、「個人専用」性質と矛盾しない (他人に開く経路ではない)。

---

## いる / いらない (2026-05-21 決定)

| 機能 | 判定 | 理由 | Issue |
|---|---|---|---|
| Calendar (個人の複数カレンダー) | いる | 時間軸の背骨。共有レイヤーは剥がす | — |
| Todos / Memos / Tags | いる | タスク・フリーノート・インデックス | — |
| Table Memo | いる (実装済) | フォーマットリフィルそのもの | KONBU-15 (Done) / KONBU-16 (CLI/MCP) |
| 横断検索 | いる | 一冊から引く動作 = コア UX | KONBU-14 |
| AI Chat | いる (エンジン) | 手帳を書く・引く執事 | KONBU-5 (high) |
| MCP / CLI | いる | 執事が手帳を操作する経路 | KONBU-16 |
| リマインダー通知 | いる | デジタル化の付加価値 | KONBU-3 (Done) |
| Export / Import | いる | 手帳 = 自分の資産、所有権の担保 | — |
| 認証 / 設定 / i18n | いる | 土台 | — |
| **Public Shares** | いらない | 個人専用と正面衝突 | KONBU-17 |
| **Published Resources** | いらない | 同上 | KONBU-17 |
| **マルチユーザーカレンダー / outbound feed** | いらない | 外に漏らす経路 | KONBU-18 |
| **Tools (ブックマーク)** | いらない | 手帳メタファー上もっとも弱い | KONBU-19 |

### 追加 (デジタル化 / inbound)

| 機能 | 軸との関係 | Issue |
|---|---|---|
| Google カレンダー取り込み表示 | 一冊集約 / inbound | KONBU-20 |
| デイリービュー (日付集約) | スケジュール背骨 / デイリーリフィル | KONBU-21 |

### 検討して見送り

- **アドレス帳 / 連絡先** — 紙の手帳では定番だったが、スマホ (iOS/Google Contacts) が既に一元管理。手帳アプリが 2 つ目の連絡先 DB を持っても二重メンテで死ぬ。**不採用**。
- **デイリー"ページ" (新エンティティ)** — データはメモで足りる。価値はデータ型ではなく集約ビュー側にある (→ KONBU-21)。
- **iCal エクスポート (konbu → 外部カレンダー)** — 所有権/バックアップは `export json` が担保済み。これは「konbu の予定を別カレンダーアプリに流す」interop-out であり、konbu を sink にする方向 (Google は取り込むが流し出さない) と逆 + 実用途なし。**不採用** (KONBU-18 で outbound feed と共に撤去。将来必要なら token feed ではなく単独ローカルコマンドとして再追加)。

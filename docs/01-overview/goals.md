# 目的・解決する課題

> **Status**: Active | 最終更新: 2026-03-17

本ドキュメントは、konbuの目的と解決する課題を明確にする。

---

## プロジェクトの目的

konbuの目的は、個人に散在する情報を1つの空間に統合し、書き出した情報を保持・更新・再アクセスできる軽快なワークスペースを提供することである。加えて、AI が代理人としてその空間を操作できるようにし、整理・検索・添削・操作の手間を減らす。セルフホスト版ではデータの完全な所有権を、クラウド版ではセットアップ不要の手軽さを提供する。

---

## 解決する課題

### 課題一覧

| # | 課題 | 影響 | 優先度 |
|---|------|------|--------|
| 1 | 情報ツールの分散 | 情報が複数サービスに散らばり、横断的に探せない。どこにあるかを覚える必要がある | High |
| 2 | Markdownだけでは表現しきれない | メモは書けても、予定やToDoを無理に同じUIで扱うと不自然になる | High |
| 3 | データの非所有 | 外部サービスにデータがロックインされ、エクスポートが困難 | High |
| 4 | 既存ツールの重さ | Notion等のオールインワンツールは起動・操作が重い | Medium |
| 5 | API非対応 | CLIやスクリプトからデータにアクセスできない | Medium |
| 6 | 書き出した情報の維持コスト | 情報を一箇所に集めても、整理・検索・書き直し・実行は都度自分でこなす必要がある | Medium |

### 課題の詳細

#### 課題1: 情報ツールの分散

| 項目 | 内容 |
|------|------|
| 現状 | メモはObsidian、ToDoはTickTick、カレンダーはGoogle Calendar等、複数サービスを併用 |
| 問題点 | 「どこに書いたか」を思い出せない。横断検索ができず、頭の中に位置情報を保持し続ける必要がある |
| 解決策 | メモ・ToDo・カレンダー・リンクを1つの情報空間に統合し、pg_trgmによる横断全文検索を提供 |

#### 課題2: Markdownだけでは表現しきれない

| 項目 | 内容 |
|------|------|
| 現状 | Markdownは強力だが、予定やToDoをそのまま押し込むと表現も操作も不自然になる |
| 問題点 | 情報は1つなのに、UIが情報の種類に合っていないため、入力も再利用も効率が落ちる |
| 解決策 | text-first を維持しつつ、予定・ToDo・テーブルには専用UIを与え、同じ情報空間の中で扱う |

#### 課題3: データの非所有

| 項目 | 内容 |
|------|------|
| 現状 | データが外部サービスのサーバーに保存されている |
| 問題点 | サービス終了時にデータが失われるリスク。エクスポート形式が限られる |
| 解決策 | セルフホストでPostgreSQLに保存。JSON/Markdownエクスポートでいつでも持ち出し可能 |

#### 課題6: 書き出した情報の維持コスト

| 項目 | 内容 |
|------|------|
| 現状 | 情報を一箇所に集めても、検索、整理、添削、変換、更新の細かな作業は手動で残りがち |
| 問題点 | 情報を一箇所に置けても、維持や再整理の手間が残ると日常の思考を邪魔する |
| 解決策 | AIが情報空間にアクセスできる前提を用意し、自然言語から検索・整理・添削・操作を代理実行できるようにする |

## 成功基準

| 基準 | 指標 | 目標値 |
|------|------|--------|
| 日常利用 | ブラウザのスタートページとして毎日使う | 毎日 |
| 応答速度 | ページ遷移・検索の体感速度 | 200ms以内 |
| データポータビリティ | 全データをJSON/Markdownで取得可能 | 100% |
| CLI対応 | 全リソースのCRUDがCLIで完結 | 全エンドポイント対応 |
| 再発見性 | 「どこにあるか」を覚えずに情報へ到達できる | 日常利用で成立 |
| 思考負荷 | 雑事や位置記憶ではなく、本来の思考に集中できる | 主観的に改善を実感 |
| 代理実行性 | AIに任せたい整理・検索・添削・操作が自然言語で成立する | 日常利用で成立 |

## 非目標

以下は本プロジェクトの目標ではない：

| 非目標 | 理由 |
|--------|------|
| リアルタイム共同編集 | 個人ワークスペースであり、マルチユーザー同時編集は不要 |
| WYSIWYGエディタ | Markdown編集とプレビュー分離を設計方針とする |
| プラグインシステム | 機能が少ないことは弱みではなく設計判断。過剰に作らない |
| エンタープライズ向け機能 | RBAC、監査ログ、SAML/OIDC等は対象外 |

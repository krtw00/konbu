# Konbu — Personal Workspace

> 更新: 2026-03-13
> ステータス: OSS + クラウド版に向けた設計

---

## コンセプト

**「ブラウザを開いたらここから始まる、自分専用のワークスペース」**

- Notion/Obsidian 的な「セカンドブレイン」を、軽量・セルフホスト可能な形で提供
- クラウド版（SaaS）とセルフホスト版（OSS）の両方を展開
- 情報の一元化と検索性に徹底的にこだわる

### ポジショニング

| 軸 | konbu | Notion | Obsidian |
|----|-------|--------|----------|
| デプロイ | セルフホスト + クラウド | クラウドのみ | ローカルのみ |
| 重さ | 軽量 | 重い | 軽量 |
| ポータル機能 | あり（ツールランチャー） | なし | なし |
| 検索 | 全文 + ベクトル横断 | 全文のみ | ローカル全文 |
| 日本語 | ファースト | 対応 | 対応 |
| 料金 | セルフホスト=全機能無料 | フリーミアム | コア無料 |

## ビジネスモデル（Open Core）

```
セルフホスト (OSS / MIT)         クラウド版 (SaaS)
──────────────────────         ──────────────────────
全機能無料                       Free tier + 有料プラン
自分でDB・サーバー管理            マネージド
開発者・技術者向け               一般ユーザーも対象
```

### 課金の切り口（案）

| 軸 | Free | Pro |
|----|------|-----|
| 容量 | メモ100件、ツール10個 | 無制限 |
| ベクトル検索 | テキスト検索のみ | AI類似検索付き |
| API | read-only | フルAPI |
| エクスポート | JSON | Markdown/PDF |
| ユーザー数 | 1人 | チーム・共有 |

※セルフホスト版は常に全機能利用可能。課金はクラウド版のみ。

## ロードマップ

### Phase 1: OSS公開の最低ライン

#### 1-1. 認証 + 初期セットアップ
- 自前認証（メール+パスワード）。Cloudflare Access非依存で動くこと
- OAuth（Google）対応
- 初回起動時のユーザー作成フロー

#### 1-2. マルチテナント
- DBスキーマにuser_id追加（既存テーブルへのマイグレーション）
- アプリ層でのデータ分離

#### 1-3. ユーザー設定UI
- テーマ選択（既存）
- カレンダー開始曜日
- 言語切替
- 設定画面の新設

#### 1-4. i18n
- 日本語 / 英語の2言語対応
- 英語デフォルト（クラウド版前提）

### Phase 2: プロダクト価値の強化

#### 2-1. ダッシュボード（ホーム画面）
- 今日の予定
- 未完了タスク
- 最近のメモ
- カスタマイズ可能なウィジェット配置

#### 2-2. メモ強化
- **画像・ファイル添付**（S3 or ローカルストレージ）
- **Markdownプレビュー**（編集モード ↔ プレビューモードの明確な切替。WYSIWYGは採用しない）
- **メモ間リンク** `[[メモ名]]`
- テーブル型メモ（構造化データ / JSONB）

#### 2-3. 横断検索
- 専用APIエンドポイント（メモ・Todo・イベント横断）
- 全文検索（pg_bigm）
- ベクトル類似検索（pgvector）— 結果は「関連」として1段下げて表示

#### 2-4. カレンダー改善
- 日曜始まりをデフォルト（設定で変更可能）
- 曜日色分け（土曜=青、日曜=赤）
- 日本の祝日対応
- 繰り返しイベント
- リマインダー / ブラウザ通知

#### 2-5. ツール強化
- カテゴリ / グルーピング
- ヘルスチェック（URL疎通確認）

### Phase 3: エコシステム

#### 3-1. Webクリッパー
- ブラウザ拡張でURL・テキスト・画像を即保存
- コマンドパレットからの即入力

#### 3-2. インポート / エクスポート
- Notion（Markdown export対応）
- Obsidian Vault（Markdown + frontmatter）
- Google Calendar（iCal）
- JSON / Markdownエクスポート

#### 3-3. クラウド版インフラ
- プラン管理（Free / Pro）
- Stripe連携
- ランディングページ

### Phase 4: 将来拡張（検討段階）

- AI連携（要約・生成 — Pro機能候補）
- プラグイン / 拡張API
- チーム・共有機能
- モバイルアプリ（PWA強化）

### フロントエンド選定理由（ADR）

Vanilla JS → React への移行を決定。

- **AI支援品質**: Reactはトレーニングデータ量最大。AIコーディング中心の開発に最適
- **エコシステム**: ライブラリ選択肢が最も豊富。将来の機能拡張で詰まらない
- **TypeScript統合**: TSX標準で型推論が強力
- **shadcn/ui**: コピー方式でコンポーネント完全制御。依存が軽い
- **実績**: 同チームのデュエログプロジェクトで同一スタック（React + shadcn/ui + Zustand）を運用済み

## 設計原則

1. **REST API がデータ操作の唯一の窓口** — Web UI も CLI も外部連携も同じ API を通る
2. **入力の摩擦ゼロ** — 思いついた瞬間に書き始められる速度
3. **検索は最重要機能** — 「あれどこに書いたっけ」をゼロにする
4. **Markdownは編集とプレビューを明確に分離** — WYSIWYGは採用しない
5. **セルフホストで全機能動作** — クラウド版の制限はあくまで容量・付加価値のみ
6. **過剰な機能を入れない** — グループウェア的機能（ワークフロー、設備予約等）は対象外
